## 一、产品概述

- **定位：** 对话式 AI 助手，30 秒内根据位置/人数/状态生成个性化周末出行方案
- **目标用户：** 一二线城市 20-35 岁年轻人
- **MVP 覆盖城市：** 苏州 + 上海 + 杭州
- **交付形态：** 微信小程序（评审决议，原 PRD 为独立 APP）

### 核心流程

用户发起对话 → Agent 最多问 3 个问题（城市/人数/状态）→ 15 秒内出 2 个方案卡片 → 用户选"就这个"

### 成功指标

- 方案采纳率 > 40%
- 对话轮数 < 5 轮
- 周回访率 > 30%

## 二、用户痛点

| 痛点 | 典型用户声音 |
| --- | --- |
| 🧠 决策瘫痪 | "刷了很多攻略，最后决定躺一天" |
| 📍 地域不匹配 | "可是我在小县城" |
| 😴 精力错配 | "上班想出去，周末只想躺" |
| 🧩 信息碎片化 | "要多久？能推车吗？有停车场吗？" |
| 🎭 场景不匹配 | 独处/约会/亲子需求完全不同 |

### 用户画像

| 人群 | 核心痛点 |
| --- | --- |
| 独居打工人 | 周末不知道干嘛，刷手机一天就没了 |
| 情侣 | 约会没新意，想找互动性强的活动 |
| 带娃家长 | 需要亲子友好、安全、低消费的去处 |
| 小城市用户 | "可是我在小县城"——热门攻略全是大城市的 |

## 三、功能清单（MVP Scope）

### P0 必须有

| 功能 | 描述 |
| --- | --- |
| 智能提问 | 最多3个问题收集上下文，支持一句话多信息解析 |
| 天气查询 | 自动查目标城市周末天气，影响推荐 |
| 方案生成 | 基于小红书数据，输出2个方案卡片 |
| 方案详情 | 选定后展开：路线站点、时间、花费、tips |

| 拒绝重推 | 不喜欢可以说原因，换一批 |
| 特殊标签 | 孕妇友好/亲子/无障碍/免费等标签筛选 |

| 查看原帖 | 链接到小红书原帖 |

## 四、方案卡片信息结构

```
方案卡片 {
  标题: "双塔市集赏花散步"
  一句话描述: "玉兰花季，吃喝逛一条线"
  emoji图标: 🌸
  基础信息 { 预计时长, 人均花费(粗粒度区间), 交通方式, 适合人数, 距离市中心 }
  标签[] { ✅孕妇友好, ✅人少, ✅免费, ✅有餐饮 }
  路线站点[] {
    站点 { 名称, 建议到达时间, 停留时间, 推荐内容, 导航链接, 步行到下一站距离 }
  }
  贴心提示[] { 天气穿衣, 路况, 卫生间位置 }
  数据来源[] { 帖子标题, 赞数, 链接 }
}
```

## 五、对话设计规范

**语气：** 像靠谱的本地朋友，简洁有趣，emoji 适度

**示例：**

- 标准流程："周末好！📍在哪？👥几人？⚡想动还是躺？"
- 一句话跳过："苏州，带3岁娃，人少公园" → 直接出方案
- 拒绝后追问原因，调整重推

**禁止：** ❌ 超过2个方案 ❌ 长篇大论 ❌ 商业广告 ❌ 恶劣天气推户外 ❌ 催促决定

### 边界场景处理

| 场景 | 处理方式 |
| --- | --- |
| 天气恶劣（暴雨/台风） | 推荐室内方案（博物馆/商场/咖啡馆），提示天气预警 |
| 节假日（五一/国庆等） | 提示人流量大，优先推荐小众/冷门地点 |
| 周日晚上发起 | 提示"本周末快结束了"，推荐短时轻量活动或顺延到下周 |
| 用户说"随便" | 基于默认值（当前城市、2人、中等精力）直接出方案 |
| 用户指定城市不在 MVP 范围 | 告知暂只支持苏州/上海/杭州，引导选择 |
| 用户需求矛盾（如"想躺又想爬山"） | 追问确认优先级，或出一动一静两个方案 |
| 深夜使用（23:00-06:00） | 正常响应，但默认推荐次日方案 |

## 5.5 埋点事件清单（MVP）

| 事件 | 触发条件 | 用途 |
| --- | --- | --- |
| 方案卡片曝光 | 2个方案卡片展示给用户 | 计算采纳率分母 |
| 点击"就这个" | 用户选定方案 | 强采纳，核心指标 |
| 卡片展开/查看详情 | 用户点击卡片查看完整路线 | 兴趣度衡量 |
| 30s 无操作离开 | 卡片曝光后 30s 内无交互且离开 | 隐性拒绝 |
| 点击"都不喜欢，换一批" | 用户主动拒绝 | 方案质量反馈 |

## 六、技术依赖

| 能力 | 用途 | 备选方案 |
| --- | --- | --- |
| 小红书数据 | 获取热帖和攻略内容 | 自建爬虫 / 现有 skill |
| 天气 API | 查询目标城市天气 | wttr.in / 和风天气 |
| 地图 API | 生成导航链接、计算步行距离 | 高德 / 百度地图 |
| LLM | 理解用户意图、生成方案文案 | DeepSeek / Qwen（评审会建议） |
| 地理编码 | 城市名 → 坐标 | 高德地理编码 API |

### 性能优化方向

| 环节 | 当前耗时（估） | 优化方向 |
| --- | --- | --- |
| 小红书实时搜索 + 抓帖子内容 | ~90s | 每日预缓存，查询时直接读缓存 |
| 天气 API 查询 | ~5s | 可并行，问题不大 |
| 地理编码/距离计算 | ~10s | 可并行 |
| LLM 生成方案文案（串行调 2 次） | ~60s | 并行生成 2 个方案 + streaming |
| 整体串行等待 | ~剩余 | 所有 API 并行化 |

## 九、风险与缓解

| 风险 | 缓解措施 |
| --- | --- |
| 小红书反爬/封号 | 缓存热门内容、多账号轮换、控制频率；Plan B：LLM 基于通用知识推荐 |

| 用户量低 | 朋友圈/小红书推广获种子用户 |
