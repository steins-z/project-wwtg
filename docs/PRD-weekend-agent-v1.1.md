## 一、产品概述

### 1.1 一句话描述

一个对话式 AI 助手，30 秒内根据用户的位置、人数、状态和偏好，生成个性化的周末出行方案。

### 1.2 要解决的问题

用户在周末出行决策中面临的核心痛点（来源：小红书评论区调研）：

| 痛点 | 用户原话 | 严重程度 |
|------|---------|---------|
| 决策瘫痪 | "刷了很多攻略，最后决定躺一天" | ⭐⭐⭐⭐⭐ |
| 地域不匹配 | "可是我在小县城" | ⭐⭐⭐⭐ |
| 精力错配 | "上班想出去，周末只想躺" | ⭐⭐⭐⭐ |
| 信息碎片化 | "要多久？能推车吗？有停车场吗？" | ⭐⭐⭐ |
| 场景不匹配 | 独处/约会/亲子需求完全不同 | ⭐⭐⭐ |

### 1.3 目标用户（MVP）

**主要用户：** 一二线城市 20-35 岁年轻人（打工人、情侣、新手父母）  
**核心场景：** 周五晚上 ~ 周六上午，在想"这个周末干嘛"的时刻

### 1.4 成功指标（MVP 阶段）

- 方案采纳率 > 40%（用户看完方案后点"就这个"）
- 单次对话轮数 < 5 轮（含确认）
- 用户回访率 > 30%（下个周末还来问）

### 2.3 用户拒绝方案 → 换一批流程

```
用户点击 [都不喜欢，换一批]
        ↓
┌─────────────────────────────────────────┐
│  Agent 追问（1个问题）                   │
│                                         │
│  "没关系！能告诉我哪里不满意吗？"        │
│                                         │
│  [太远了] [不想户外] [想吃东西为主]      │
│  [想更刺激的] [预算太高] [其他]          │
└──────────────┬──────────────────────────┘
               ↓
      根据反馈调整搜索条件
      生成新的 2 个方案
               ↓
      如果连续拒绝 3 次：
      "看来今天的推荐都不太对味 😅
       要不你告诉我心里大概想去什么类型的地方？
       或者直接说个关键词，我帮你搜"
               ↓
      切换为「搜索模式」而非「推荐模式」
```

### 2.4 边界场景处理

| 场景 | 处理方式 |
|------|---------|
| 用户城市搜不到内容 | "这个城市的攻略比较少，我帮你搜了附近的{大城市}周边游，有些可以当天往返" |
| 天气恶劣（暴雨/极端高温） | 自动切换室内方案："明天有大雨🌧️ 推荐室内活动" |
| 用户有特殊限制（孕妇/轮椅/老人） | 标注 ✅/❌ 适合标签，优先筛选无障碍、平坦路线 |
| 节假日（非普通周末） | 提醒"本周是清明假期，热门景点人会多，推荐避开" |
| 用户说"随便" | 按低决策成本原则推荐：离家近 + 免费 + 时间短的方案 |
| 已经是周日晚上 | "这周末快结束啦，要不我帮你先收藏下周的方案？周五再提醒你" |
| 用户只是闲聊不是真的要出门 | 识别意图，回复"收到，想出门的时候随时找我！" |

## 五、对话设计规范

### 5.1 语气

- **像一个靠谱的本地朋友**，不是客服不是导游
- 简洁有趣，不说废话
- 用 emoji 但不过度

### 5.2 示例对话

**场景1：标准流程**

> 👤 周末去哪玩  
> 🤖 周末好！三个小问题：  
> 📍 在哪个城市？  
> 👥 几个人出动？  
> ⚡ 想活动一下还是躺平充电？  

> 👤 杭州 和女朋友 想动一动  
> 🤖 杭州明天 12-20°C ☀️ 好天气！给你俩找了两个方案...

**场景2：一句话说清楚**

> 👤 苏州，带3岁娃，想找个人少的公园  
> 🤖 收到！苏州+带娃+人少公园，给你找...（跳过提问直接出方案）

**场景3：拒绝后**

> 👤 都不太想去  
> 🤖 没关系！是觉得太远了、不想户外、还是想换个类型？  
> 👤 想找个有意思的室内活动  
> 🤖 室内！看看这两个...

**场景4：模糊需求**

> 👤 好无聊啊  
> 🤖 无聊退散！你现在在哪个城市？我给你找点有意思的事做 😎

### 5.3 禁止事项

- ❌ 不输出超过2个方案（选择越多越纠结）
- ❌ 不写长篇大论（卡片式呈现，关键信息一目了然）
- ❌ 不推荐明显的商业广告内容
- ❌ 不在恶劣天气推荐户外（安全第一）
- ❌ 不催促用户做决定

### 5.4 埋点事件清单（MVP）

| 事件 | 触发条件 | 用途 |
|------|---------|------|
| 方案卡片曝光 | 2 个方案卡片展示给用户 | 计算采纳率分母 |
| 点击"就这个"（强采纳） | 用户选定方案 + 查看详情 | 核心指标 |
| 弱采纳 | 用户点"就这个"但没看详情 | 区分兴趣层级 |
| 卡片展开/查看详情 | 用户点击卡片查看完整路线 | 兴趣度衡量 |
| 30s 无操作离开 | 卡片曝光后 30s 内无交互且离开 | 隐性拒绝/流失 |
| 点击"都不喜欢，换一批" | 用户主动拒绝当前方案 | 方案质量反馈 |

**采纳率定义：** 40% 指标使用"强采纳"口径（点"就这个" + 查看详情）

## 七、交付形态

**评审会决策：微信小程序**

MVP 阶段使用微信小程序验证，原因：
- 开发周期 4-6 周（vs 独立 APP 8-12 周）
- 微信生态获客容易，分享卡片天然支持
- 卡片 UI 体验可做到位
- 数据跑通后再决定是否做独立 APP

### 登录方式
- MVP 阶段：**微信授权登录**，不要求手机号（减少漏斗流失）
- 后续迭代：做偏好学习/周五推送时再加手机号绑定

## 九、风险与缓解

| 风险 | 影响 | 缓解措施 |
|------|------|---------|
| 小红书反爬/封号 | 数据源断裂 | 缓存热门内容、多账号轮换、控制请求频率；**Plan B：爬虫不可用时退化为 LLM 基于通用知识推荐 + 用户行后反馈迭代** |
| 推荐不准确 | 用户流失 | MVP阶段限定几个城市精细运营 |
| 小红书内容版权 | 法律风险 | 只引用摘要+链接回原帖，不搬运全文 |
| 天气数据不准 | 推荐失误 | 用多数据源交叉验证 |
| 用户量低 | 无法验证 | 先在朋友圈/小红书自身推广获种子用户 |

## 十一、实测 Demo 验证

### Demo 场景

**输入：** "苏州，和老公，我是孕妇不能去人太多的地方"（截图见 `demo-screenshot.png`）

### Demo 验证了什么

| 设计意图 | Demo 表现 | 验证结果 |
