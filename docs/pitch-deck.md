## Part 1：我们要做什么

### 一个观察

小红书已经成为中国年轻人的"生活决策引擎"。不知道吃什么、穿什么、去哪玩、送什么礼物——第一反应是打开小红书搜一搜。

**但搜到了 ≠ 解决了。**

小红书擅长提供信息（"这10个地方值得去"），却不擅长帮用户做决定（"根据你的情况，你应该去这个"）。

这个 gap，恰好是 AI Agent 最擅长的事：**理解个人上下文 → 整合多源信息 → 给出个性化决策。**

### 我们的计划

建立一套**持续运转的需求挖掘机制**：

```
小红书热帖/评论区
      ↓  挖掘
识别"决策型需求"（用户在纠结、在求推荐、在选择困难）
      ↓  评估
筛选 Agent 能比人做得更好的需求
      ↓  验证
快速搭建 Agent 原型，用真实用户测试
      ↓  迭代
验证成功 → 产品化；失败 → 换下一个需求
```

**不是做一个产品，而是建立一条"从需求到 Agent 产品"的流水线。**

### Agent 可行性评估

我们用 4 个维度打分（1-5），评估每个需求是否适合用 Agent 解决：

- **决策复杂度**：需要综合多少信息
- **Agent 优势**：相比自己搜，Agent 能好多少

## Part 3：为什么第一个做"周末去哪玩"

### 三个原因

**① 综合得分最高（17分）**

在所有需求中，"周末去哪玩"在频次、复杂度、Agent优势、数据可得性四个维度的综合表现最好。

**② Agent 优势最明显（5/5）**

对比自己刷小红书 vs Agent 帮你决定：

| 维度 | 自己刷小红书 | Agent |
|------|------------|-------|
| 时间 | 15-30 分钟 | **30 秒** |
| 决策 | 越看越纠结 | **直接给 1-2 个** |
| 地域 | 看运气 | **基于你的位置** |
| 天气 | 自己查 | **自动排除不合适的** |
| 精力匹配 | 没有 | **问你状态再推荐** |
| 实操细节 | 自己翻评论区 | **自动整合** |

这个需求天然需要**整合多源信息**（位置 + 天气 + 人数 + 偏好 + 攻略内容），这正是 Agent 相比搜索引擎的核心差异。

**③ 评论区验证了痛点真实存在**

我们深入读了 Top 帖子的评论区，用户自己说出了痛点：

> _"刷了很多攻略，最后发现决定躺一天 😂"_ — 江苏  
> _"可是我在小县城 😭"_ — 重庆  
> _"上班的时候想出去玩，真到周末就只想躺着"_ — 评论区  
> _"全程多少时间？""能推婴儿车吗？""附近有吃的吗？"_ — 大量实操提问

**核心洞察：用户不缺信息，缺的是"根据我的情况帮我做决定"。**

### 为什么不先做其他的？

| 需求 | 没有首选的原因 |
|------|--------------|
| 今天吃什么（16分） | 频次虽高但决策太轻，用户容易"随便吃点"解决 |
| 旅行攻略（16分） | 频次太低（一年几次），不适合做 MVP 验证留存 |
| 不知道穿什么（15分） | 数据可得性较低——需要用户衣橱信息，冷启动难 |
| 送什么礼物（13分） | 事件驱动、频次低 |
| 求推荐App（10分） | Agent 优势不明显，人工推荐更可信 |

## Part 4：产品方案 —「周末搭子」

### 一句话定位

> **30 秒内，根据你在哪、和谁、什么状态，给你一个"直接能去"的周末方案。**

### 核心体验流程

```
用户："周末去哪玩" / "好无聊" / "明天干嘛"
        ↓
Agent 快速收集上下文（最多3问，支持一句话说清）
  📍 在哪  👥 几个人  ⚡ 想动还是想静
        ↓
后台：查天气 + 搜小红书 + 综合筛选（< 15秒）
        ↓
输出 2 个方案卡片（不是10个——减少选择负担）
  ┌────────────────────────┐
  │ 🌸 方案A（推荐）        │
  │ 双塔市集赏花散步        │
  │ ⏱半天 💰人均50 🚇地铁可达│
  │ ✅孕妇友好 ✅人少        │
  └────────────────────────┘
        ↓
用户选定 → 生成出行计划
  路线 + 每站导航 + 建议时间 + 天气提醒 + Tips
        ↓
不满意 → 说原因 → 换一批（最多3轮后切搜索模式）
```

### 实测 Demo

我们已经跑通了一次完整流程（截图见附件 `demo-screenshot.png`）：

**输入：** "苏州，和老公，我是孕妇不能去人太多的地方"  
**Agent 动作：** 查天气（7-15°C 多云）→ 搜小红书 3 组关键词 → 读 3 篇详情帖  

**输出亮点：**
- 方案 A「双塔市集赏花散步」— 4 站路线，细到推荐"老客满蛋饼"、"slight coffee 草莓 dirty"
- 方案 B「何山公园森林散氧」— 贴心提醒"蚊虫多带防蚊喷雾"、"没有售卖点提前备水"
- 两个方案均标注 ✅孕妇友好、预估花费、步行距离
- 用户只说了一句话，Agent 提取了 4 个关键信息（苏州、两人、孕妇、避人多）

**用户评价：** "效果很好" ✅  
**总耗时：** 约 3 分钟（含天气查询 + 小红书搜索 + 帖子详情读取）

> 💡 **这个 demo 证明了核心假设：Agent 可以从"一句话"出发，输出"直接能去"的完整方案。**

### 已确认决策

| 决策项 | 结论 |
|--------|------|
| 产品名 | 周末搭子 |
| 场景 | MVP 只做周末 |
| 城市 | 苏州 + 上海 + 杭州 |
| 商业模式 | MVP 免费 |
| 数据策略 | 每日缓存 + 实时兜底 |

### MVP 里程碑（4周）

| 周 | 产出 |
|----|------|
| W1 | 确定技术方案 + 搭建对话流 |
| W2 | 接入数据源 + 方案生成逻辑 |
| W3 | 内测（10 个种子用户） |
| W4 | 优化 + 正式发布 |

### 成功指标

| 指标 | 目标 |
|------|------|
| 方案采纳率 | > 40% |
| 单次对话轮数 | < 5 轮 |
| 周回访率 | > 30% |

## Part 5：竞品分析

### 市场上没有人专门做这件事

| 竞品 | 做了什么 | 没做什么 |
|------|---------|---------|
| **携程 AI 助手** | 长途旅游行程规划 + 预订 | 不懂你周末的状态，不问"累不累" |
| **高德 AI 探索** | 基于位置推荐附近 POI | 只给点位，不给完整路线方案 |
| **小红书搜索** | 海量 UGC 攻略 | 越看越纠结，不帮你做决定 |
| **豆包/通用 AI** | 什么都能聊 | 没有本地实时数据，推荐质量差 |

### 我们的差异化

| 维度 | 他们 | 我们 |
|------|------|------|
| 信息量 | 给你更多选择 | **帮你做决定（只出 2 个）** |
| 输出 | POI 列表 / 帖子列表 | **完整出行卡片（路线+导航+Tips）** |
| 个性化 | 千人一面 | **≤3 问理解你当下状态** |
| 场景 | 全场景平台 | **All-in 周末，做到极致** |

> 一句话：**大厂在做"AI + 旅游平台"，我们在做"AI 帮你做周末决定"。**
